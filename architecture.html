1. DNS resolution cez Cloudflare
WEDOS deleguje mysak.fun na Cloudflare nameservery. Cloudflare vrátí svou vlastní Edge IP (ne přímo IP backendu) — provoz jde vždy přes CF proxy (orange cloud).
2. Cloudflare Access interceptuje požadavek
Pokud URL nemá platnou CF_Authorization cookie, Access přesměruje na login page. Session trvá 24h, poté je nutné znovu přihlášení.
3. OAuth 2.0 flow s Entra ID (Azure AD)
Cloudflare přesměruje na login.microsoftonline.com s parametry App Registration (client_id, tenant_id = f50acfeb-...). Uživatel se přihlásí pomocí michal.burdik@gmail.com. Entra vrátí JWT.
4. Kontrola policy
Access ověří, zda JWT email odpovídá whitelist pravidlu: include { email = ["michal.burdik@gmail.com"] }. Pokud ano, nastaví CF_Authorization cookie.
5. Proxied request na backend
Cloudflare přepošle požadavek na backend (AWS EIP / Azure LB / ALB). Přidá hlavičky Cf-Access-Authenticated-User-Email a Cf-Access-Jwt-Assertion — backend může JWT verifikovat.
ℹ️
auto_redirect_to_identity = true — všechny Access aplikace mají zapnutý automatický redirect na Entra ID přihlašovací stránku. Uživatel nikdy neuvidí Cloudflare Access "login" stránku — rovnou skočí na Microsoft login.

Workflow — jak nasadit změnu
Přidat/změnit DNS záznam nebo Access policy
Edituj main.tf, spusť terraform plan, zkontroluj diff, pak terraform apply. Většina změn se projeví do 5 minut (CF cache).
Změna nginx-ingress IP (az-seip)
Zjisti novou IP: terraform output nginx_ingress_ip v azure-seip/infra. Aktualizuj var.azure_seip_nginx_ip v .env nebo prd.tfvars, pak terraform apply.
Přidání nového uživatele do Access
Přidej email do všech cloudflare_access_policy bloků: include { email = ["michal.burdik@gmail.com", "novy@example.com"] }, pak terraform apply.
Nová chráněná služba
1. Přidej cloudflare_record (DNS). 2. Přidej cloudflare_access_application s allowed_idps = [cloudflare_access_identity_provider.entra_id.id]. 3. Přidej cloudflare_access_policy. 4. terraform apply.
